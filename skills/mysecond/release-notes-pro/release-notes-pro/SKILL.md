# Release Notes Pro

Write release notes users actually read — benefit-first and engaging.

## When to Use This Skill
- Any feature release (big or small)
- Writing changelog entries
- In-app announcements and emails

## What You'll Get

I'll generate 4 versions of your release notes, optimized for each channel:
- **Changelog version** — Structured with benefit headline, what's new, and how-to steps
- **Email version** — Conversational with clear CTA
- **In-app banner** — Ultra-brief for notifications
- **Social post** — Shareable announcement (for major releases)

## What You'll Need

**Critical inputs (ask if not provided):**
- What feature or change are you announcing?
- What launch tier is this? (Major, Medium, or Minor)
  - **Major (T1):** Big feature worth celebrating — gets all 4 formats
  - **Medium (T2):** Helpful improvement — gets changelog + email
  - **Minor (T3):** Small fix — just changelog

**Nice-to-have inputs:**
- Specific metrics or improvements to highlight
- Who's affected (all users vs specific segments)
- Related help docs or how-to resources

## Process

### Step 1: Review Your Context

I'll start by checking your context files to understand:
- **product.md** — What's this feature? What problem does it solve?
- **personas.md** — Who benefits most? What language do they use?
- **company.md** — Brand voice and product name

I'll share what I find and use it to frame the benefits. For example:
> "I found 'Resource Planning v2' in your product.md as a recent launch. Your Jordan persona (PM) was frustrated by 'manual workload tracking.' I'll write the release notes emphasizing how this saves them time."

### Step 2: Confirm Release Details

If I need more context, I'll ask:
> "Before I write these release notes, I need:
> 1. What feature are you announcing? (I found [X] in product.md — is that it?)
> 2. What tier is this? (Major, Medium, or Minor)
>
> I can also pull benefit framing from your personas if relevant."

I won't write release notes with placeholder benefits — I'll get the real value prop first.

### Step 3: Write Benefit-First

I'll lead with what users can DO now, not just what you built:
- ✅ "Find files 3x faster"
- ❌ "Added new search functionality"

### Step 4: Match Tone to Tier

I'll adjust the tone based on launch significance:
- **Major:** Exciting, celebratory, detailed (all 4 formats)
- **Medium:** Informative, helpful (changelog + email)
- **Minor:** Brief, matter-of-fact (changelog only)

### Step 5: Include How-To Steps

I'll show users how to actually use the new feature — not just announce it.

### Step 6: Format for Each Channel

I'll optimize each version:
- **Changelog:** Structured, scannable sections
- **Email:** Conversational with one clear message
- **In-app:** Ultra-brief, action-oriented
- **Social:** Shareable with hashtags (if Major)

## Output Template

```markdown
# Release Notes: [Feature Name]

**Release Date:** [Date]
**Version:** [X.Y.Z]
**Tier:** Major / Medium / Minor

## Context
*What I found in your files:*
- **Feature:** [From product.md]
- **Benefit framing:** [From personas.md — what pain this solves]
- **Target persona:** [From personas.md]

---

## Changelog Version

### [Benefit-First Headline]

[One paragraph describing what users can now do and why it matters.]

**What's New:**
- [Specific change 1]
- [Specific change 2]

**How to Use It:**
1. [Step 1]
2. [Step 2]
3. [Step 3]

**Availability:** [Who has access / rollout plan]

---

## Email Version

**Subject Line:** [Benefit-first, <50 characters]

## In-App Banner Version

## Social Version (if Major)

[2-3 sentences about the benefit]

[Link]

```

## Examples

### Feature-First (❌ Don't)
> "We added a new dashboard with filtering capabilities."

### Benefit-First (✅ Do)
> "Find the data you need in seconds. Our new dashboard lets you filter by date, team, and status — no more scrolling through endless lists."

## Framework Reference
**Benefit-first communication**:
- Lead with what users can DO, not what you BUILT
- Show, don't just tell
- Match tone to significance

## Tips for Best Results

1. **Use your context files** — I'll pull benefit framing from personas, feature context from product.md
2. **Match persona language** — I'll use the words your users use, not internal jargon
3. **Lead with the outcome** — "Save 2 hours/week" beats "New dashboard feature"
4. **Tier determines length** — Major releases get all 4 formats, Minor just needs changelog
5. **Include how-to** — I'll show how to use it, not just announce it
